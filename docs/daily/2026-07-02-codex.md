## Key Updates

- **LangChain's July 1 cluster pushed durable agent context into the harness layer**: [OpenWiki](https://www.langchain.com/blog/introducing-openwiki-an-open-source-agent-for-repo-documentation) turns repo documentation into an agent-maintained, CLI-generated knowledge surface that coding agents can retrieve instead of stuffing whole codebases into context. Paired with [RLMs in Deep Agents](https://www.langchain.com/blog/how-to-use-rlms-in-deep-agents), the direction is clear: long-running agents need structured memory, code-driven recursion, and traceable updates rather than bigger chat histories.

- **Production evals are moving from final-answer scoring to trajectory replay**: LangChain and Harbor's [agent eval stack](https://www.langchain.com/blog/unified-stack-for-evaluating-agents) framed long-running, stateful agents as needing runners that preserve state, replay trajectories, and connect Deep Agents, LangSmith Sandboxes, and experiments. The latest [LangGraph 1.2.7 release](https://github.com/langchain-ai/langgraph/releases) also stayed focused on reliability fixes around snapshot/overwrite semantics, JSON roundtrips, and dependency hygiene.

- **Agent observability became a sharper vendor and architecture debate**: Laminar's July 1 [agent observability ranking](https://laminar.sh/article/top-6-agent-observability-platforms) argued that agent traces are materially different from single-call LLM logs, while LangChain's [Pendo/Novus case study](https://www.langchain.com/blog/how-pendo-used-langsmith-to-trace-novus-from-user-behavior-to-code-fixes) showed the operational version: tracing product-agent behavior all the way from user issues to code fixes and PM-reviewed evals.

- **FDE crossed from AI-lab pattern to hyperscaler operating model**: AWS announced a [$1B Forward Deployed Engineering organization](https://www.aboutamazon.com/news/aws/aws-1-billion-forward-deployed-ai-engineers) to embed engineers with customers and co-develop agentic AI systems. The high-signal part is not the budget; it is AWS promising that every engagement leaves behind knowledge graphs, runbooks, architectural docs, trained internal champions, and systems inside the customer's governance boundary.

## Hot Discussions

- **MCP moved from integration story to security and audit boundary**: Microsoft Incident Response's [MCP tool poisoning analysis](https://www.microsoft.com/en-us/security/blog/2026/06/30/securing-ai-agents-ai-tools-move-from-reading-acting/) made the core risk concrete: a tool-description update can act like a hidden system-prompt change once agents move from reading to acting. Obot's [MCP observability guide](https://obot.ai/blog/mcp-observability-how-to-monitor-ai-agent-activity-in-the-enterprise/) captured the corresponding control-plane requirement: identity, server inventory, tool-call parameters, policy decisions, sensitive-data handling, latency, and audit trails must be visible as one chain.

- **Agent-native DevOps is becoming an application-platform primitive**: Vercel's [Agent guide](https://vercel.com/kb/guide/vercel-agent) packages PR review, production-anomaly investigation, project Q&A, and code changes around deployment/runtime context, secure sandboxes, validation, access controls, approval gates, and credits. This supports the broader trend from "AI coding assistant" to "agent with deploy context, rollback surface, and governed action budget."

## Actionable Insight

- For the next internal production-agent pilot, treat the harness as the product: maintain an agent-readable wiki or knowledge graph with provenance, instrument full trajectories with OpenTelemetry-compatible spans, gate tool access through identity and policy logs, run trajectory replay evals before release, and make FDE-style delivery leave behind runbooks, eval datasets, and ownership transfer rather than only a working demo.
